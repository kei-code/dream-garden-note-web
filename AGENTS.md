# AGENTS.md

## コミュニケーション方針

- ユーザーは忖度のない客観的な意見を求めている。
- 品質を上げるため、問題点・懸念点・弱い仮説は率直に伝える。
- 回答では「事実」「推測」「意見」をできるだけ分ける。
- 分からないことは分からないと言い、根拠のない断定を避ける。
- デザインや事業判断では、見栄えだけでなく目的達成・販売可能性・実装現実性も評価する。

## 実装開始前の確認ルール

- 実装・ファイル編集・新規作成に入る前に、原則としてユーザーの了承を得る。
- ユーザーが明確に「実装して」「作って」「修正して」などと指示した場合は、実装に進んでよい。
- 調査、比較、テキスト案、設計案、懸念点の提示は、実装前の確認なしで行ってよい。
- 実装に入る場合は、何を編集するかを短く説明してから着手する。

## 開発判断の原則

- 最優先は「絵本として読む体験」。Webサイトらしい説明や装飾より、作品への没入感を優先する。
- まず大枠を固める。ページ構成、PC/スマホの読書体験、作品一覧、販売導線を優先し、細部の演出は品質フェーズで詰める。
- ただし、ページめくり演出は作品体験の核なので、後で必ず作り込む重要項目として扱う。
- チープなリアル風より、まずは控えめで邪魔しない仮演出を優先する。
- 決定済みのデザイン仕様は `docs/design_decisions.md` に残し、判断がぶれないようにする。
- 開発手順、優先順位、フェーズ、保留事項は `docs/development_policy.md` に残す。

## Python 環境管理

- Python パッケージのインストールには必ず `uv` を使う。
- `pip install` は使わない。代わりに `uv add` を使う。
- スクリプト実行は `uv run python` で行う。
- CLI ツールのインストールは `uv tool install` を使う。

## 環境汚染の懸念があるときのルール

以下に該当する操作を行う前には、必ず日本語で分かりやすく説明し、ユーザーに確認を取る。

- グローバル環境へのパッケージ・ツールのインストール（pip, npm -g, brew など）
- システムの設定ファイルや環境変数の変更
- 複数プロジェクトに影響しうる依存関係の変更
- バージョンの上書き・ダウングレード

確認時は以下の形式を使う。

```text
環境への影響があります

【何をしようとしているか】
〇〇をインストールしようとしています。

【影響範囲】
グローバル環境 / このプロジェクトのみ

【環境汚染のリスク】
〇〇という理由で、他のプロジェクトに影響する可能性があります。

【推奨する代替手段】
uv を使って分離された環境にインストールすることを推奨します。

実行してよいですか？
```
